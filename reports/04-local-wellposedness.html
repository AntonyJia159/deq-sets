Set-DEQ Notes · #4

Going Local: Does a Radius-Bounded Equilibrium Even Converge?

Zhibai Jia · 21 June 2026 · working notes, not for circulation beyond reviewers

1. Why we changed the model

Notes #1–#3 studied a set-DEQ where every point talks to every other point through a global pool (mean or attention). That model is mostly unimodal: the equilibrium does little that a single forward pass couldn't, so there was no headroom for the interesting equilibrium phenomena we were chasing. This note changes one thing: each point now interacts only with nearby points — those inside a ball of radius r in feature space — instead of the whole set.

That single change is motivated by a unification we arrived at on paper. Three properties we care about — train-small/infer-large (size generalization), cheap deletion (edit-locality), and decentralized/federated solving — all turn out to be consequences of one architectural commitment: local interaction. If a point only listens to its neighbors, then (a) the rule is the same regardless of how many points there are, (b) removing a far-away point can't ripple across the set, and (c) the computation can be split across machines that only exchange boundary information. Locality is the root; the three properties are its readouts.

But before any of that pays off, one question gates everything: does a local equilibrium even converge to a well-defined answer? This note is that gate test, run with no training — pure random-weight networks — because convergence and uniqueness are properties of the architecture, not of any learned task. If the iteration doesn't settle from random weights, no amount of training will save it.

2. The knobs, in plain language

There are a lot of dials in this experiment, and it's easy to lose track. Here is every one, what it physically means, and why we vary it.

chain		 The shape of the point cloud, to stress-test convergence on different structures: points spread evenly, points in 3 tight blobs, or points strung along a line (where information has to travel many hops).

And the four things we measure

  • conv% (convergence rate) — the fraction of sets where the iteration actually settled (residual below 1e-5) within 200 steps. This is the headline: did it reach an answer at all?
  • iters — how many steps it took, when it did converge.
  • fp_gap (fixed-point gap) — uniqueness. We solve the same set from several random starting points and measure how far apart the answers are. 0.000 = always the same equilibrium (one basin). Large = different starts land in different places (multistability, multiple basins).
  • osc% (oscillation rate) — the fraction of sets where the residual was still going up near the end, i.e. the iteration was bouncing rather than settling. This is the specific failure we worried about for the latent graph.

3. The first run failed — and the failure was the lesson

The naive implementation defines a neighbor with a hard cutoff: point j is either in the ball or out of it, a yes/no switch at distance r.

neighbor(i, j) = 1 if ‖p_i − p_j‖ < r else 0

With the fixed (input) graph, this converged beautifully — residual down to 1e-7. But with the state-dependent (latent) graph, it refused to converge: the residual got stuck around 1e-3 and bounced there forever. Here is the side-by-side on one representative set:

graph_sourceresidual @ 50@ 100@ 200@ 300
input (fixed)1.6e−48.9e−78.8e−88.4e−8
latent (state-dependent)8.6e−41.3e−38.2e−42.1e−3

The reason is exactly the thing that makes the latent graph interesting. As the internal state moves, points drift across the ball boundary — and with a hard cutoff, a neighbor blinks on and off from one iteration to the next. Each blink discontinuously changes what every nearby point sees, so the update never holds still. The map we are iterating is discontinuous in its own state, and a discontinuous map has no reason to have a fixed point a solver can find. Damping (taking smaller steps) softened the bouncing but never removed it — it plateaued around 1e-3 no matter what.

4. The fix: a soft neighborhood — and why it's the whole point

Replace the yes/no switch with a smooth fade: a point is "mostly a neighbor" just inside the ball and "mostly not" just outside, with a gentle transition of width τ.

weight(i, j) = sigmoid( (r − ‖p_i − p_j‖) / τ )

Now when a point drifts across the boundary, its influence changes continuously from, say, 0.6 to 0.4 instead of snapping 1→0. The whole update becomes a continuous function of the state, the iteration can settle, and it does — immediately and completely:

τgraph_sourceresidual @ 100@ 200@ 300
2.0latent6.4e−78.8e−88.6e−8
0.5latent7.9e−78.9e−88.9e−8
0.1latent1.5e−68.8e−88.3e−8

Even at τ = 0.1 — an almost-hard edge — the state-dependent graph now matches the fixed graph's 1e-7 convergence. The smoothness, not the sharpness, is what mattered.

The clean takeaway. Continuity of the neighbor kernel is what makes a state-dependent local equilibrium well-posed. This is almost certainly why the regime was empty in the literature: the obvious implementation (a hard ball) appears not to converge, so it looks like a dead end — unless you realize the discontinuity, not the locality, is the culprit, and smooth the boundary. One line of code separates "doesn't work" from "works."

5. The full sweep

With the soft kernel in place, we ran the whole grid — both aggregators, both graph sources, six radii, four set sizes, three geometries, two seeds, with no training. Averaged over geometries and seeds, two regimes emerge cleanly.

5a. local_mean: well-posed everywhere

The mean-pool aggregator is the boring, reassuring case. Across every radius and set size, fixed and state-dependent alike, it converges and lands on a single unique fixed point (fp_gap essentially zero). State-dependence is indistinguishable from the fixed graph — the soft kernel fully closes the gap.

The only softening is at large radius and large N on the chain geometry (convergence dipping to ~79%), and crucially that dip is identical for the fixed graph — so it's ordinary weak contraction at scale, not a state-dependence problem. The sweet spot is a moderate radius (r = 2–4).

5b. local_attn + latent: genuine multistability — the interesting cell

The attention aggregator with a state-dependent graph is where something new appears. At moderate radius it still converges most of the time, but it lands in different basins from different starts — fp_gap of 0.2–0.5, i.e. real multistability.

graph_sourcerNconv%fp_gaposc%
input (fixed)2.06484%0.0006.7%

Read the last row carefully: at r = 100 (everyone is a neighbor) the multistability vanishes — fp_gap collapses back to zero, because that's just the global-attention model, which we already knew is nearly unique. The multistability is a creature of locality. Restricting attention to a moderate-radius neighborhood manufactures coexisting equilibria that the global model does not have. This is the modern-Hopfield "attention = associative memory" picture made geometric: local attention carves the state space into basins.

Whether that multistability is a feature or a bug depends on what we build next. For an associative-memory / basin-steering story (deletion nudges the equilibrium between basins) it is the feature. For a single-answer regression task it would be a liability to be tamed (smaller radius, contraction, or the trainable path-independence recipe from Note #2). Either way, it is new structure — the empty cell was not empty for lack of interest.

6. Verdict and what it unlocks

Gate: GREEN. A local, state-dependent set-equilibrium converges to a well-defined answer, provided the neighbor kernel is smooth. Mean-pool is unique everywhere; local attention converges but is multistable at moderate radius — and that multistability is created by the locality itself.

This was the precondition for everything downstream. With the gate passed, the next experiments become meaningful:

  • Size generalization — train at small N, test at large N; check the equilibrium reaches answers a fixed-depth network can't, and that iterations-to-converge scale with the set's diameter (information traveling one hop per step).
  • Edit-locality — delete a point and measure how far the equilibrium shifts as a function of graph distance from the deletion. The hoped-for result: the shift decays with distance, so re-converging after an edit costs work proportional to the neighborhood, not the whole set.
  • Federated solve — split one set across "sites," solve by exchanging only boundary latents, and confirm it lands on the same equilibrium as the centralized solve.

A caveat we already see coming (and will measure, not assume): the federated efficiency depends on the partition respecting the geometry. If two sites' points are spatially interleaved, the "boundary" is the whole set and nothing is saved. Local solving buys communication savings exactly when the data is locally clustered — which is also the only regime where local interaction is the right model in the first place. That self-consistency is reassuring, but it is a scope condition we'll state plainly rather than paper over.
